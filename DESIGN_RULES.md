# SleepAI 调研 Deck 设计规则

本规则适用于 `/Users/oliver/Downloads/SleepAI/SleepAI商业计划&市场调研` 下所有新增或重做的 HTML 调研幻灯片、首页与可分享展示页。目标是让它们像同一个网站，而不是一组彼此独立的文件。

## 1. 适用范围与目录

- 对外展示页统一放在 `reports/<date>-<slug>/index.html`
- 原始文字稿统一保留在 `sources/md` 或 `sources/notes`
- 共享样式与脚本只放在 `assets/css` 与 `assets/js`
- 新 deck 默认从现有共享壳层出发，不再重新写一套完全独立的 UI

## 2. 信息架构

- 每份 deck 都必须有清晰封面、核心判断、关键证据、行动建议与收束结论
- 首页 `index.html` 只做入口与导航，不重复展开完整报告内容
- 首页不放制作过程、维护说明、研究方法、协作提示等内部信息
- deck 内文只放结论、证据和含义，不放研究过程、搜索过程、推理过程
- 若有附注、方法、详细来源，优先放在页脚、来源区或 md 源稿，不占正文主叙事
- 如果某组量化证据或案例已经被主 deck 吸收，就不要再单独挂一个重复的附加 deck 入口

## 3. 文件命名

- 目录名统一用 `YYYY-MM-DD-topic-slug`
- 主文件统一叫 `index.html`
- 源稿 md 文件名与 report slug 尽量一一对应，方便页脚回链
- 图像、附件、补充素材放在各自 report 子目录或共享 `assets`

## 4. 视觉主题

- 全站统一使用深色夜间基调，不为单个 deck 随意换成浅色主主题
- 主色只保留一组核心信号：`mint/cyan` 作为强调，`soft blue` 作为辅助
- 背景保持“深色渐变 + 轻网格/光斑”语言，避免纯黑纯色平铺
- 卡片统一使用半透明深色面板、细线框、高斯感阴影，不做厚重拟物
- 首页列表默认纵向单列，不做横向双列卡片墙
- 关键信息列表默认优先纵向堆叠，避免为了“丰富”强行做左右不均的双栏

## 5. 设计 Token

- 优先复用 `assets/css/deck.css` 与 `assets/css/site.css` 中的变量
- 常用最大宽度保持在 `1160px-1180px`
- 圆角保持在 `22px-24px`
- 全站统一使用 `MiSans / MiSans Global` 作为主要字体
- 新 deck 若必须写内联样式，也要沿用同一套 token 命名
- 链接必须显式定义 `default / visited / hover / active` 四种状态，禁止出现浏览器默认蓝紫色
- 字号层级遵循克制规则：Hero 约 `40-60px`，Section title 约 `28-38px`，Big number 约 `28-42px`，Body 约 `16px`
- 不使用夸张的超大字做“气势”；优先参考 Apple 与 Google 官方网页那种收敛、清晰、留白充足的层级
- 结尾判断与摘要区不得出现“大字极大、说明极小”的失衡层级；标题与补充说明要保持可连续阅读

## 6. 顶部导航与站点感

- 每份 deck 顶部都必须有 `返回总览` 入口
- 左上角品牌区统一由 `分类 kicker + deck 标题` 组成
- 顶部导航只保留 5-8 个最关键章节，避免过密
- 右上角统一保留一个轻交互入口，优先使用 `专注模式`

## 7. 内容密度

- 每屏只讲一个判断，不把多层逻辑堆在一页
- 每个 section 优先 1 个主标题、1 段结论、1 组图表或 2-4 个要点
- 文本以短句为主，避免大段正文
- 同一屏元素数量过多时，优先删字，不优先缩小字号
- 禁止“这是一个入口页”“点击查看”“适合分享”“建议未来”这类无信息增量的废话
- 禁止“现在可以拍板什么”“高置信动作”“一句话判断”“复制摘要”这类口语化、机器翻译感或无专业价值的栏目命名

## 8. 图表规则

- 图表首先服务“判断”，不是服务“展示全面”
- 一张图只承载一个问题，例如份额、对比、优先级、路线图
- 默认使用条形、堆叠条、指标卡、矩阵、时间线，不滥用复杂图形
- 图表颜色遵循统一语义：主结论用 accent，次要项用 muted，风险项用 warn/danger
- 图表旁必须有一句含义解释，不只放数字
- 数字卡、案例卡、指标块必须先保证容器内的留白和换行安全；不允许出现数字、标题或标签逼近边缘的窒息布局

## 9. 组件规则

- 统一复用：`topbar`、`hero`、`card`、`chip/tag`、`section-head`、`deck-footer`
- 页脚统一包含：站点名 + 当前 deck 名 + 源稿 md 链接
- 需要附加说明时优先用 `support-note`、`ghost`、静态来源区，不要把说明挤进主标题区
- 新组件必须先判断能否复用现有模式，避免 deck 与 deck 之间出现完全不同的控件语言
- 信息密度较高的案例区禁止使用翻面卡；优先使用静态卡片，把关键信息一次展示完整

## 10. 交互规则

- 交互要轻量，默认只做切换与聚焦；不要加入对阅读主流程帮助不大的功能
- 不做重 JS 动画，不做复杂滚动特效，不做花哨但无信息价值的转场
- 所有交互在无脚本时最好仍能阅读主要内容
- 必须兼容桌面投屏与手机浏览，不做只适合宽屏的布局
- 手机端必须单列、可点按、无遮挡、无横向滚动；桌面端必须保持清晰层级和留白

## 11. 来源与证据呈现

- 正文只展示必要结论，不把完整参考列表塞进主叙事
- 原始来源尽量放在 md 源稿；HTML 里保留简版来源区即可，若来源数量不多则默认直接展开显示
- 每个关键数字最好能在相邻文案中看出时间和口径
- 若数字为估算，必须在 md 源稿中可追溯，HTML 中可简写为“估算”或“推算”

## 12. 中文化规则

- 面向团队内部版本默认中文优先
- 品牌名、产品名、少数行业缩写可保留英文，例如 `Web`、`App`、`PWA`
- 如保留英文术语，应确保中文读者不看英文也能理解结论
- 按钮、导航、图表标签、提示语优先全部中文
- 若使用 MiSans，需在项目中明确注明使用了 MiSans Fonts

## 13. GitHub Pages 友好规范

- 所有链接优先使用相对路径，保证仓库直接部署可用
- 避免依赖本地绝对路径、构建工具输出或私有资源
- 每个 report 目录都应能独立打开 `index.html`
- 首页必须能稳定链接到所有 report 与源稿目录

## 14. 发布前检查清单

- 顶部是否有返回总览入口
- 页脚是否有源稿 md 回链
- 视觉 token 是否仍与全站一致
- 手机宽度下是否没有明显溢出或拥挤
- 大标题、大数字、长单词、百分比与中英混排是否都没有超出容器
- 章节导航、按钮、抽屉、切换器是否可用
- 文字是否以中文为主，英文是否只保留必要项
- 是否只保留结论与证据，没有研究过程说明

## 15. 新 Deck 推荐流程

1. 先写 md 源稿，明确结论与证据。
2. 复制最接近的现有 report 作为 HTML 起点，不从零造壳。
3. 优先复用共享样式与已有组件。
4. 完成后补页脚回链与首页入口。
5. 通过发布前检查清单后再纳入 `reports/`。
