# Pufflo Brain Playbook · Codex Rules

本文件只适用于：

- `reports/2026-03-19-pufflo-brain-playbook/`

不适用于：

- 其他调研幻灯片
- 其他报告页
- 首页 `index.html`
- 共享视觉规范的普遍规则

## 1. 页面定位

- 这是 `Pufflo 产品入口与 AI 行为执行手册`
- 主读者是产品、前端、后端、模型工程同事
- 优先保证“可查阅、可执行、可直接对照”，不追求短
- 页面可以很长，只要目录清晰、锚点稳定、信息不丢

## 2. 不可删改的底层原则

- 原始 prompt 的中英双语正文、few-shot、边界句，不得为了“更简洁”而压缩、合并或删减
- 页面里不能出现制作过程、组织思路、作者解释、脑内 OS
- 禁止写这类话：
  - `这一章`
  - `这里要讲`
  - `先看`
  - `最关心`
  - `这份手册按`
  - `阅读顺序`
  - `总结如下`
  - `用于说明`
- 不放截图进页面，也不把截图资源放进站点
- 页面中的产品字段层、交互层、路由层，不能替代 prompt 层；两者要并存

## 3. 什么必须写进 Prompt

只有会影响 LLM 输出结果的规则，才写进对应模块的 `prompt-block`。

必须写进 prompt 的内容包括：

- 会改变模型输出语气、结构、长度、节奏的规则
- 会改变模型选择哪种表达方式的规则
- 会改变模型 fallback 行为的规则
- 会改变模型何时停止、何时转轻、何时不再解释的规则
- 会改变模型是否要分块输出、如何分块输出的规则
- 会改变模型是否可轻量过渡到下一个模块的规则
- 会改变模型能否引用、如何引用、不能伪造什么的规则
- 会改变 push 文案生成的约束、边界、记忆使用方式的规则

判断方法：

- 如果这条要求会让 LLM 输出的文字不同，就应写进 prompt
- 如果这条要求只影响页面怎么摆、按钮怎么放、播放器怎么浮、内容怎么展示，就不写进 prompt

## 4. 什么只留在文稿，不写进 Prompt

以下内容默认只留在产品规格 / 文稿层，不进入 prompt：

- `点赞 / 分享`
- `悬浮播放器`
- `首页卡片位置`
- `页面层级与入口排序`
- `按钮位置`
- `视觉样式`
- `经典引用样式` 的纯前端视觉实现
- `页面该怎么排版`
- `哪些内容放左边、哪些放右边`

判断方法：

- 如果这是前端、客户端、交互、视觉的事，就不要硬塞给 LLM

## 5. 例外：文稿规则需要同步进 Prompt 的典型场景

以下几类虽然一开始看像产品要求，但如果会影响模型输出，必须同步写进 prompt：

- `经典引用样式`
  - 不是把视觉样式写进 prompt
  - 而是要求模型把 `Quote` 与 `Reflection` 分开输出
  - 这样前端才有能力做单独展示

- `转入睡眠监测`
  - 如果模型负责说出那句过渡话，就要把“何时可转入、怎么转入”写进 prompt
  - 如果只是客户端自动跳转，而模型不负责说话，则可以只留在文稿里

- `下午 3 点主动关心`
  - 如果是模型写 push 文案，就必须把记忆使用边界、禁止伪熟悉、禁止复述旧对话写进 prompt

## 6. 当前页面的 Prompt 设计重点

### Story

- 用户说 `太有剧情 / 太清醒 / 太重` 时，prompt 必须要求模型立刻收轻
- 背景声只能托住内容，不能抢叙述
- 用户已经放松时，允许一句轻量过渡到睡眠监测或静音延续

### Meditation

- 中段如果用户已安静，prompt 必须要求模型减少解释、减少步骤、减少选择题
- 背景声要极轻、无歌词、低存在感
- 可以在合适时机轻量转入睡眠监测或背景延续

### Breathing

- 输出结构固定：`开场白 -> 呼吸循环 -> 结束词`
- 如果用户讨厌计数，prompt 必须要求模型立刻切无计数版本
- 稳定后可轻量过渡，不继续说教

### Music

- 规则推荐时只给一个最佳方向，不给列表
- 用户进入 `只要背景、不要说话` 后，说一句就停
- 输出要落到明确的 ambience / soundscape 标签，不能只写“放点音乐”
- 如果音乐是其他模块后半段延续，不重新拉用户做选择

### Religion

- 若输出经典内容，必须分成 `Quote` / `Reflection`
- 没有准确可用引文时，必须明确退回 `reflection / prayerful wording`
- 引文要克制，不能扩展成说教
- 用户已放松时，可轻量转入睡眠监测或静音延续

### Push

- push 文案是单条短文案，不是小段解释
- `3:00 PM` 主动关心只适用于非首次用户且有真实记忆
- 不能假装熟悉、不能逐字复述旧对话、不能像监视
- 如果记忆不足，就退回普通温和关心

## 7. 文案格式规则

- 面向用户可见的双语文案，默认格式是：
  - `English first`
  - `中文翻译放在后面的括号里`

标准写法：

- `How is your afternoon going?（今天下午过得怎么样？）`

不要写成：

- 先中文后英文
- 英文和中文拆成两行但没有清晰对应
- 同一页面混用多种双语顺序

## 8. 中文表达规则

- 中文必须自然、专业、可直接给产品和工程同事读
- 少用机器翻译腔和半中半英堆叠
- 不要写管理层空话
- 不要写“像说明书的说明书”
- 标题、导语、表格说明都只写内容本身，不写作者组织意图

## 9. 导航与信息结构规则

- 页面可长，但锚点必须稳定
- `模块执行规则` 必须保留细颗粒度子项，方便跳转
- 产品字段层要放在 prompt 层前面
- 页面不能只剩 prompt；也不能只剩产品文稿

## 10. 改这个页面前的检查清单

- 这条新要求会改变模型输出吗？
  - 会：写进对应 prompt
  - 不会：只放文稿

- 这条规则是输出结构要求，还是视觉要求？
  - 输出结构要求：写进 prompt
  - 视觉要求：只留前端/文稿

- 新增文案是否保持 `英文在前，中文括号在后`？

- 是否误删了原始 bilingual prompt 或 few-shot？

- 是否又写进了脑内 OS？

- 是否把 UI 规则误塞进 prompt？

- 是否把真正应进 prompt 的规则只停留在 prose？
